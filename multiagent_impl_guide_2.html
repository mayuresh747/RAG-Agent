OVERVIEW

What We Are Building

What you already have (do not change these)

Your existing system does the following in sequence:

User query
→
gpt-4o-mini classifier (SIMPLE/COMPLEX)
→
300 regex patterns → pick libraries
→
Embed query (text-embedding-3-large)
→
ChromaDB vector search (25 candidates/library)
→
Cosine rerank → top_k=12 or 24
→
Build prompt → gpt-5.1 → stream

This works for single-agency factual questions. It breaks for multi-agency conflict analysis because it treats all agencies the same and cannot guarantee balanced evidence across both sides of a friction pair.

→
Your existing gpt-5.1 → stream

STEP 1 OF 7

Query Analyzer

What this step does and why

Your current classifier asks one question: is this COMPLEX or SIMPLE? That determines top_k (24 or 12) and nothing else. The library router then independently runs 300 regex patterns to pick collections.

This step replaces both with a single LLM call that returns structured JSON answering four questions simultaneously:

  1. What mode is this query — factual (A), named-agency conflict (B), topic conflict (C), or full audit (D)?
  2. Which agencies are relevant — and which ChromaDB collection does each map to?
  3. How relevant is each agency to THIS specific query — score 1, 2, or 3?
  4. How many total chunks do we need in the final answer?

retrieval/analyzer.py — create this file

Create the folder retrieval/ at your project root, then create this file inside it.

Key things to notice in the code below:
— response_format={"type": "json_object"} forces the LLM to return valid JSON. Never skip this.
— temperature=0 makes routing deterministic. You do not want random agency selection.
— The prompt explains the relevance scoring rules in plain English because the LLM needs to understand your corpus, not just return a number.

retrieval/analyzer.py
import json
from openai import AsyncOpenAI
from config import COLLECTION_MAP, LLM_FAST

oai = AsyncOpenAI()

# ─────────────────────────────────────────────────────────────────────────────
# ANALYZER PROMPT
#
# This prompt does the work of both your binary classifier AND your
# 300-regex library router — in a single call that understands meaning,
# not just keywords.
#
# Every instruction below is deliberate:
#   - Mode rules prevent the LLM from over-classifying simple questions as D
#   - Relevance rules are query-specific, not agency-specific
#   - top_k rules give enough headroom for the reranker to work
#   - Agency guidance prevents COURT from being missed on legal disputes
# ─────────────────────────────────────────────────────────────────────────────

ANALYZER_PROMPT = """
You are a query router for a regulatory RAG system. You have 8 document
collections covering Washington State and Seattle regulations:

  RCW        — Revised Code of Washington (state statute)
  WAC        — Washington Administrative Code (state regulations)
  EXEC_ORDER — Washington Governor Executive Orders
  IBC_WA     — International Building Code as adopted by Washington
  SMC        — Seattle Municipal Code (city ordinance)
  DIR        — Seattle Director's Rules (city admin interpretations)
  SPU        — Seattle Public Utilities Design Standards
  COURT      — Washington Supreme/Appellate Court opinions

Your job: analyze the user's query and return ONLY valid JSON with
these exact fields — no explanation, no markdown, no extra text.

{
  "mode": "A" | "B" | "C" | "D",
  "agencies_in_scope": ["AGENCY", ...],
  "agency_relevance": {"AGENCY": 1 | 2 | 3, ...},
  "top_k": ,
  "requires_numerical_comparison": true | false
}

─── MODE RULES ───────────────────────────────────────────────────────────────

A = FACTUAL: User wants to know what one set of rules says.
    No comparison, no conflict analysis. Single agency focus.
    Example: "What does SMC say about ADU height limits?"

B = NAMED CONFLICT: User explicitly names 2 or 3 agencies.
    Look for: "between DIR and IBC", "compare RCW and SMC", etc.
    Example: "What are the conflicts between DIR rules and IBC on egress?"

C = TOPIC CONFLICT: User wants conflicts on a topic but did NOT name agencies.
    You decide which agencies are relevant. 2-6 agencies typical.
    Example: "What EV charging conflicts should I know about?"

D = FULL AUDIT: User describes a development scenario and wants all risks.
    Trigger words: "audit", "all risks", "full analysis", "everything I need".
    Example: "I'm building a 6-story mixed-use in SODO, what are all my risks?"

─── AGENCY RELEVANCE SCORING (per query, not per agency) ────────────────────

3 = Primary: This agency is the main source of rules for this query.
    The question is primarily ABOUT this agency's domain.

2 = Supporting: This agency likely has relevant provisions but is not
    the main focus. Needed to check for conflict or supporting context.

1 = Completeness: Include only because there might be something there.
    Low confidence it's relevant. Included for thoroughness in C/D modes.

─── TOP_K RULES ─────────────────────────────────────────────────────────────

A (1 agency):        top_k = 12
B (2 agencies):      top_k = 16
B (3 agencies):      top_k = 20
C (3-4 agencies):    top_k = 24
C (5-6 agencies):    top_k = 32
D (all agencies):    top_k = 40

─── AGENCY SELECTION GUIDANCE ───────────────────────────────────────────────

ALWAYS include COURT if query mentions any of:
  court, lawsuit, ruling, nexus, constitutional, invalidated, exaction,
  takings, challenged, overturned, legal precedent, appellate

ALWAYS include IBC_WA if query mentions any of:
  building code, egress, fire, structural, occupancy, accessible,
  ADA, construction, height, setback (building-specific)

ALWAYS include SPU if query mentions any of:
  water main, sewer, drainage, stormwater, utility connection,
  infrastructure, pipe, SPU, public utilities

ALWAYS include EXEC_ORDER if query mentions any of:
  governor, executive order, emergency directive, state emergency

For general Seattle development questions: always include SMC + DIR + RCW.
For state-level regulatory questions: always include RCW + WAC.
"""

async def analyze_query(query: str) -> dict:
    """
    Analyzes a user query and returns structured routing metadata.

    Returns dict with keys:
      mode               : "A" | "B" | "C" | "D"
      agencies_in_scope  : list of agency labels e.g. ["RCW", "SMC", "COURT"]
      agency_relevance   : dict of agency → 1|2|3
      top_k              : int, target final chunk count
      requires_numerical_comparison : bool
      collections        : dict of agency → ChromaDB collection name (added here)

    Raises:
      json.JSONDecodeError if LLM returns malformed JSON (rare at temp=0)
      KeyError if LLM returns unknown agency label (caught in planner)
    """
    resp = await oai.chat.completions.create(
        model=LLM_FAST,             # gpt-4o-mini — cheap, fast, deterministic
        messages=[
            {"role": "system", "content": ANALYZER_PROMPT},
            {"role": "user",   "content": query},
        ],
        response_format={"type": "json_object"},  # enforces valid JSON output
        temperature=0,              # deterministic routing
        max_tokens=300,             # JSON is small; cap to avoid wasted tokens
    )

    result = json.loads(resp.choices[0].message.content)

    # Resolve agency labels → your actual ChromaDB collection names
    # Only include agencies that exist in COLLECTION_MAP (safety filter)
    result["collections"] = {
        ag: COLLECTION_MAP[ag]
        for ag in result["agencies_in_scope"]
        if ag in COLLECTION_MAP
    }

    return result

STEP 2 OF 7

Query Planner

What this step does and why

The planner takes the analyzer's JSON output and converts it into a list of concrete retrieval tasks — one task per agency. Each task specifies exactly how many candidates to fetch from that agency's ChromaDB collection.

This is where the three-factor weighting system is applied:

  1. Corpus size weight — WAC has 4,300 files. To have a fair chance of finding the right chunk, you need to fetch proportionally more candidates from it than from SPU which has 14 files.
  2. Query relevance score — from the analyzer. An agency with relevance=3 needs more candidates because we must have enough text from it to quote in the friction matrix.
  3. Floor — the minimum candidates per agency regardless of the math, to ensure no agency gets accidentally zeroed out by low weights.
Budget calculation — worked example

Using the water main query: Mode C, 4 agencies, top_k=28.

Total candidate pool = top_k × 3 = 84 candidates. The 3× multiplier gives the reranker enough material to work with.

Raw weight per agency = corpus_weight × relevance_score

COURT1.534.528%24824
RCW2.024.025%21621
SPU1.033.019%16516

COURT gets the most candidates (24) because it has relevance=3 AND there is a direct court case about permit exactions. RCW gets 21 because it has a large corpus. SPU gets 16 despite a small corpus because relevance=3. SMC hits its floor of 11 (the formula gave 11, which exceeds the floor of 6).

agents/planner.py
from dataclasses import dataclass
from config import CORPUS_WEIGHT, AUTHORITY_RANK

# ─────────────────────────────────────────────────────────────────────────────
# AGENCY TASK
# A dataclass is used instead of a plain dict to catch typos early
# (e.g. task.agensy raises AttributeError, task["agensy"] returns None)
# ─────────────────────────────────────────────────────────────────────────────

@dataclass
class AgencyTask:
    agency:       str    # canonical label e.g. "RCW"
    collection:   str    # ChromaDB collection name e.g. "rcw_chapters"
    query:        str    # the search query to run against this collection
    n_candidates: int    # how many raw results to fetch before reranking
    relevance:    int    # 1|2|3 — kept here for use in collector's floor logic
    boost_numbers: bool  # True if query involves fees/dimensions/thresholds

# ─────────────────────────────────────────────────────────────────────────────
# CONFLICT FLOORS
# Minimum candidates to fetch in conflict modes (B / C / D) regardless
# of weight calculation.
# Rationale:
#   COURT=8  — opinions are long, dense, and surface poorly in vector search.
#              Need extra candidates to reliably find relevant precedent.
#   RCW=6    — large corpus, 6 is barely enough for meaningful coverage.
#   SMC=6    — city ordinance is always needed for city conflict queries.
#   Others=4 — basic floor to avoid accidentally zeroing out small agencies.
# ─────────────────────────────────────────────────────────────────────────────

CONFLICT_FLOOR = {
    "COURT":      8,
    "RCW":        6,
    "WAC":        6,
    "SMC":        6,
    "DIR":        4,
    "IBC_WA":     4,
    "SPU":        4,
    "EXEC_ORDER": 4,
}

FACTUAL_FLOOR = 3  # Mode A — minimal floor, reranker handles the rest

def calculate_budget(
    agencies:   list[str],
    relevance:  dict[str, int],
    top_k:      int,
    mode:       str,
) -> dict[str, int]:
    """
    Returns {agency: n_candidates} — how many raw candidates to
    pull from each agency's ChromaDB collection.

    The total candidate pool is top_k * 3 (3x headroom for the reranker).
    Each agency gets a share proportional to:
        corpus_weight * query_relevance_score
    Subject to a minimum floor depending on mode.

    Parameters:
        agencies  : list of agency labels from analyzer e.g. ["RCW","SMC","COURT"]
        relevance : dict from analyzer e.g. {"RCW":2, "SMC":2, "COURT":3}
        top_k     : target final chunk count from analyzer
        mode      : "A"|"B"|"C"|"D" from analyzer
    """
    is_conflict = mode in ("B", "C", "D")
    total_pool  = top_k * 3   # e.g. top_k=28 → 84 candidates to fetch total

    # Step A: raw weight per agency = corpus size × query relevance
    raw = {
        ag: CORPUS_WEIGHT.get(ag, 1.0) * relevance.get(ag, 1)
        for ag in agencies
    }
    total_weight = sum(raw.values())

    # Step B: proportional share of the candidate pool
    proportional = {
        ag: int(total_pool * (w / total_weight))
        for ag, w in raw.items()
    }

    # Step C: apply floors
    floor = CONFLICT_FLOOR if is_conflict else {}
    allocation = {
        ag: max(floor.get(ag, FACTUAL_FLOOR), proportional[ag])
        for ag in agencies
    }

    return allocation


def plan_tasks(query: str, analysis: dict) -> list[AgencyTask]:
    """
    Converts analyzer output into a list of AgencyTask objects.
    One task per agency in scope.

    Parameters:
        query    : the original user query string
        analysis : the dict returned by analyze_query()

    Returns:
        list of AgencyTask, sorted by authority rank (highest authority first)
        so logs are easier to read.
    """
    agencies    = analysis["agencies_in_scope"]
    relevance   = analysis.get("agency_relevance", {ag: 2 for ag in agencies})
    mode        = analysis["mode"]
    top_k       = analysis["top_k"]
    collections = analysis["collections"]
    boost       = analysis.get("requires_numerical_comparison", False)

    allocation  = calculate_budget(agencies, relevance, top_k, mode)

    tasks = []
    for ag in agencies:
        if ag not in collections:
            continue  # skip if collection not found
        tasks.append(AgencyTask(
            agency        = ag,
            collection    = collections[ag],
            query         = query,
            n_candidates  = allocation[ag],
            relevance     = relevance.get(ag, 1),
            boost_numbers = boost,
        ))

    # Sort by authority rank so COURT tasks appear first in logs
    tasks.sort(key=lambda t: AUTHORITY_RANK.get(t.agency, 99))
    return tasks

STEP 3 OF 7

Parallel Agency Retriever

What this step does and why

This is the core of the multi-agent design. Instead of searching one ChromaDB collection at a time, we launch all agency searches simultaneously using Python's asyncio.gather.

Think of it like sending four people to four filing cabinets at the same time instead of one person going to each cabinet one after another. Total time = slowest single search, not the sum of all searches.

Each retriever does exactly what your existing code already does against ChromaDB — a vector similarity search using the query embedding. The only difference is it runs concurrently with the others, and each result is tagged with its agency label so the collector knows where it came from.

Why the query vector is computed once and shared

All agencies use the same query embedding. Computing it once and passing it to all retrievers avoids making 4-8 identical embedding API calls. This saves cost and ~400ms of latency per search.

agents/retriever.py — create this file
agents/retriever.py
import asyncio
from openai import AsyncOpenAI
import chromadb
from config import CHROMA_PATH, EMBED_MODEL, EMBED_DIMS
from agents.planner import AgencyTask

# ─────────────────────────────────────────────────────────────────────────────
# CLIENTS
# Initialized once at module load — not inside each function call.
# Initializing chromadb.PersistentClient inside each async call causes
# file handle leaks and slows startup significantly.
# ─────────────────────────────────────────────────────────────────────────────

oai    = AsyncOpenAI()
client = chromadb.PersistentClient(path=CHROMA_PATH)


async def embed_query(text: str) -> list[float]:
    """
    Embeds the user's query into a vector.
    This is called ONCE per user query and the result is passed
    to all agency retrievers — do not call this inside retrieve_agency().

    Returns:
        list of floats — 3072 dimensions for text-embedding-3-large
    """
    resp = await oai.embeddings.create(
        input=text,
        model=EMBED_MODEL,
        dimensions=EMBED_DIMS,
    )
    return resp.data[0].embedding


def _normalize_metadata(meta: dict) -> dict:
    """
    ChromaDB stores only flat scalar values in metadata.
    If your indexer serialized lists to pipe-separated strings
    (e.g. section_ids = "RCW 82.02|RCW 82.03"), deserialize them here.

    If your metadata uses different field names, update this function.
    If a field doesn't exist, we return a safe default — never crash here.
    """
    raw_sections = meta.get("section_ids", "")
    sections = [s for s in raw_sections.split("|") if s] if raw_sections else []
    return {
        **meta,
        "section_ids": sections,
        "has_numbers": bool(meta.get("has_numbers", 0)),
        # add any other fields your indexer serialized differently
    }


async def retrieve_agency(task: AgencyTask, query_vector: list[float]) -> dict:
    """
    Searches ONE agency's ChromaDB collection and returns its results.
    This function runs concurrently with other retrieve_agency calls.

    Parameters:
        task         : AgencyTask from planner — has collection name and n_candidates
        query_vector : pre-computed embedding from embed_query()

    Returns:
        dict with keys:
            agency  : agency label e.g. "RCW"
            chunks  : list of chunk dicts, each with text, metadata, distance
                      Empty list if collection not found or error occurs.
    """
    try:
        collection = client.get_collection(task.collection)
    except Exception as e:
        # Collection doesn't exist — return empty rather than crash the whole query
        print(f"[retriever] Collection not found: {task.collection} — {e}")
        return {"agency": task.agency, "chunks": []}

    # n_results cannot exceed the collection's actual document count
    n = min(task.n_candidates, collection.count())
    if n == 0:
        return {"agency": task.agency, "chunks": []}

    results = collection.query(
        query_embeddings=[query_vector],
        n_results=n,
        include=["documents", "metadatas", "distances"],
    )

    # ChromaDB returns results nested in lists because it supports batch queries.
    # [0] unwraps the first (and only) query's results.
    chunks = []
    for cid, doc, meta, dist in zip(
        results["ids"][0],
        results["documents"][0],
        results["metadatas"][0],
        results["distances"][0],
    ):
        chunks.append({
            "chunk_id":    cid,
            "text":        doc,
            "agency":      task.agency,   # IMPORTANT: tag with agency
            "relevance":   task.relevance, # passed through for collector
            "cosine_dist": dist,           # lower = more similar
            **_normalize_metadata(meta),
        })

    return {"agency": task.agency, "chunks": chunks}


async def retrieve_all_parallel(
    tasks: list[AgencyTask],
    query_vector: list[float],
) -> list[dict]:
    """
    Runs all agency retrievals simultaneously.

    asyncio.gather fires all coroutines at once and waits for all to finish.
    Total latency = slowest single agency search (not the sum of all).
    return_exceptions=True means one failed agency doesn't crash the others.

    Parameters:
        tasks        : list of AgencyTask from plan_tasks()
        query_vector : pre-computed embedding from embed_query()

    Returns:
        list of {agency, chunks} dicts — one per successful agency.
        Failed agencies are logged and excluded from the result.
    """
    coroutines = [retrieve_agency(t, query_vector) for t in tasks]
    raw        = await asyncio.gather(*coroutines, return_exceptions=True)

    valid = []
    for i, r in enumerate(raw):
        if isinstance(r, Exception):
            print(f"[retriever] {tasks[i].agency} failed: {r}")
        elif r["chunks"]:          # exclude agencies that returned nothing
            valid.append(r)
        else:
            print(f"[retriever] {tasks[i].agency} returned 0 chunks")

    return valid

STEP 4 OF 7

Cross-Encoder Reranker

What this step does and why

Vector search finds chunks that are about similar topics as your query. The cross-encoder finds chunks that are actually useful for answering your query. These are different.

A vector search for "permit exaction nexus test" surfaces chunks about permits and about nexus — but might rank a general overview of permit types above the specific subsection that defines the nexus test standard. The cross-encoder reads the query and chunk together as a pair and scores how well the chunk answers the query. It understands that a 3-sentence subsection defining the nexus test is far more useful than a 2-page overview that mentions nexus once.

It runs after all agency candidates are pooled together — so it compares SPU chunks against RCW chunks against COURT chunks on equal footing, using the same relevance scale.

retrieval/reranker.py
from sentence_transformers import CrossEncoder

# ─────────────────────────────────────────────────────────────────────────────
# RERANKER MODEL
#
# cross-encoder/ms-marco-MiniLM-L-6-v2
#   — Downloads ~85 MB on first run, cached to ~/.cache/huggingface
#   — Outputs logit scores (can be negative) — higher = more relevant
#   — Runs entirely locally — no API call
#   — batch_size=32 is safe on CPU; increase if you have a GPU
#
# To upgrade later (better accuracy, more resources):
#   cross-encoder/ms-marco-MiniLM-L-12-v2  (~170 MB, ~2x slower, ~5% better)
# ─────────────────────────────────────────────────────────────────────────────

class LegalReranker:

    def __init__(self):
        # Load model once at startup — not per query
        self.model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

    def rerank(
        self,
        query:      str,
        candidates: list[dict],
        top_k:      int,
    ) -> list[dict]:
        """
        Scores all (query, chunk) pairs with the cross-encoder,
        sorts by score descending, returns top_k.

        Parameters:
            query      : the original user query string
            candidates : flat list of chunk dicts from all agencies
            top_k      : how many to return after scoring

        Returns:
            list of chunk dicts sorted by rerank_score (best first),
            truncated to top_k. Each chunk gets a "rerank_score" field added.
        """
        if not candidates:
            return []

        # Build (query, chunk_text) pairs — the cross-encoder reads both together
        pairs  = [(query, c["text"]) for c in candidates]

        # predict() returns a numpy array of scores, one per pair
        scores = self.model.predict(pairs, batch_size=32)

        # Attach score to each chunk and sort
        ranked = sorted(
            zip(candidates, scores),
            key=lambda x: x[1],
            reverse=True,   # higher score = more relevant
        )

        result = []
        for chunk, score in ranked[:top_k * 2]:  # keep 2× for collector's two-pass
            chunk["rerank_score"] = float(score)
            result.append(chunk)

        return result
STEP 5 OF 7

Evidence Collector

What this step does

The collector receives the full reranked pool and selects the final top_k chunks that go into the context window. For Mode A this is trivial — just take the top N. For Modes B/C/D it requires the two-pass selection to guarantee balanced evidence across conflict pairs.

The two-pass selection explained

The reranker's job is to find the globally best chunks. But "globally best" does not mean "best for building a friction matrix." A Mode C query about EV charging might produce this after pure reranking:

RankAgencyScore
1–8WAC0.85–0.72
9–16WAC0.71–0.60
17–20SMC0.55–0.50
21–28WAC0.49–0.40

WAC dominates because it has 4,300 files and many EV-related chunks. SMC and SPU appear at the bottom. But you cannot build a WAC×SMC friction pair with only 4 SMC chunks at rank 17-20. You need guaranteed representation.

Pass 1 — Reserve slots for guaranteed agencies

The floor depends on the analyzer's relevance score for this specific query. Relevance=3 agencies get 2 guaranteed slots. Relevance=2 get 1. Relevance=1 get 0. COURT always gets 2 in conflict modes regardless of relevance.

The key: guaranteed slots take the best-scoring chunks from that agency — not random ones. So even the guaranteed COURT slots are the highest-quality COURT chunks from the reranked pool.

Pass 2 — Fill remaining slots with global best

After reserved slots are filled, the remaining spots go to the globally best reranked chunks not already selected. This is where WAC's strong performance shows up — it fills most of the remaining slots on merit.

Pair coverage check

Final sanity check: for every intended conflict pair (A × B), both agencies need at least 1 chunk in the final set. If not, force in the top-scoring chunk from the missing agency. This catches edge cases where an agency's chunks all scored below the floor threshold.

from itertools import combinations
from retrieval.reranker import LegalReranker
from config import AUTHORITY_RANK

# Single reranker instance — model loaded once at startup
reranker = LegalReranker()


def _get_guaranteed_slots(agency: str, relevance: int, mode: str) -> int:
    """
    Returns the number of slots this agency is guaranteed in the final output.
    This is query-dependent (via relevance) not fixed per agency.

    Logic:
        COURT in conflict mode  → always 2 (binding precedent must appear)
        relevance 3             → 2 slots (primary agency, need enough to quote)
        relevance 2             → 1 slot (supporting, needs presence)
        relevance 1             → 0 slots (trust the reranker entirely)
        Mode A                  → 0 for all (pure reranker result)
    """
    if mode == "A":
        return 0
    if agency == "COURT":
        return 2
    if relevance == 3:
        return 2
    if relevance == 2:
        return 1
    return 0  # relevance == 1 — reranker decides


def collect_evidence(
    agency_results:   list[dict],
    query:            str,
    top_k:            int,
    mode:             str,
    agency_relevance: dict[str, int],
) -> list[dict]:
    """
    Merges, deduplicates, reranks, and selects the final top_k chunks.

    Parameters:
        agency_results   : list of {agency, chunks} from retrieve_all_parallel()
        query            : original user query string
        top_k            : target final chunk count from analyzer
        mode             : "A"|"B"|"C"|"D" from analyzer
        agency_relevance : {agency: 1|2|3} from analyzer

    Returns:
        list of chunk dicts, length <= top_k, sorted by rerank_score descending.
        Each chunk guaranteed to have: chunk_id, text, agency, rerank_score.
    """

    # ── 1. FLATTEN AND DEDUPLICATE ────────────────────────────────────────────
    # Flatten all agency results into one list.
    # Deduplicate by chunk_id — same doc can appear in multiple collection
    # results if chunk boundaries overlap across ingestion runs.
    seen, unique = set(), []
    for res in agency_results:
        for c in res["chunks"]:
            if c["chunk_id"] not in seen:
                seen.add(c["chunk_id"])
                unique.append(c)

    if not unique:
        return []

    # ── 2. GLOBAL CROSS-ENCODER RERANK ────────────────────────────────────────
    # Score all candidates against the query together.
    # top_k * 4 gives the collector enough material for two-pass selection.
    ranked = reranker.rerank(query, unique, top_k=top_k * 4)

    # Build per-agency sorted lists from the reranked pool
    by_agency: dict[str, list] = {}
    for chunk in ranked:
        by_agency.setdefault(chunk["agency"], []).append(chunk)
    # by_agency[ag] is already sorted best-first because `ranked` is sorted

    # ── 3. MODE A FAST PATH ───────────────────────────────────────────────────
    if mode == "A":
        return ranked[:top_k]

    # ── 4. TWO-PASS SELECTION (Modes B / C / D) ───────────────────────────────

    agencies_present = {r["agency"] for r in agency_results if r["chunks"]}
    used_ids = set()
    guaranteed = []

    # PASS 1: Reserve guaranteed slots per agency
    # Iterate in authority order (COURT first) so highest-authority
    # agencies get their pick before lower-authority ones.
    sorted_agencies = sorted(
        agencies_present,
        key=lambda a: AUTHORITY_RANK.get(a, 99)
    )

    for ag in sorted_agencies:
        rel    = agency_relevance.get(ag, 1)
        slots  = _get_guaranteed_slots(ag, rel, mode)
        taken  = 0
        for chunk in by_agency.get(ag, []):
            if taken >= slots:
                break
            guaranteed.append(chunk)
            used_ids.add(chunk["chunk_id"])
            taken += 1

    # PASS 2: Fill remaining slots with globally best unreserved chunks
    fill = []
    for chunk in ranked:
        if len(guaranteed) + len(fill) >= top_k:
            break
        if chunk["chunk_id"] not in used_ids:
            fill.append(chunk)

    final = guaranteed + fill

    # ── 5. PAIR COVERAGE CHECK ────────────────────────────────────────────────
    # For every intended conflict pair, both agencies need ≥1 chunk.
    # If an agency is missing entirely, force in its top-scoring chunk.
    # This catches edge cases — it should rarely fire if budget is set correctly.
    if len(agencies_present) >= 2:
        final_agencies = {c["agency"] for c in final}
        pairs = list(combinations(
            [a for a in agencies_present if a != "COURT"], 2
        ))
        for ag_a, ag_b in pairs:
            for missing in [ag_a, ag_b]:
                if missing not in final_agencies:
                    # Force in top-scoring chunk from missing agency
                    rescue = by_agency.get(missing, [])
                    if rescue:
                        final.append(rescue[0])
                        final_agencies.add(missing)
                        print(f"[collector] Pair rescue: forced in {missing}")

    # ── 6. FINAL SORT AND TRIM ────────────────────────────────────────────────
    # Re-sort by rerank_score so context block is ordered best-first.
    # Trim to top_k in case pair rescue added extra chunks.
    final.sort(key=lambda c: c.get("rerank_score", 0), reverse=True)
    return final[:top_k]

Structured Context Assembly

What this step does and why

Your current system dumps all retrieved chunks sequentially into the prompt with no structure. When you have chunks from 4-8 agencies, the LLM has to infer which chunk belongs to which agency — and frequently attributes citations incorrectly.

This step groups chunks by agency, orders groups by authority rank, and adds clear section headers. The LLM sees exactly which agency each chunk came from before it starts generating. This is what enables the VERIFY-ADJUDICATE loop to work correctly.

from config import AUTHORITY_RANK

def assemble_context(chunks: list[dict]) -> str:
    """
    Groups chunks by agency, orders groups by authority rank,
    and formats them into the structured context block that gets
    injected into the system prompt.

    The format is designed for the LLM to:
      1. Know exactly which agency each chunk came from
      2. Know the relative authority of each agency
      3. Have clear section boundaries for the VERIFY-ADJUDICATE loop

    Parameters:
        chunks : list of chunk dicts from collect_evidence()
                 Each must have: text, agency, source_file, page
                 Should have: section_ids (list)

    Returns:
        str — the formatted context block ready to append to system prompt
    """

    # ── GROUP BY AGENCY ───────────────────────────────────────────────────────
    by_agency: dict[str, list] = {}
    for c in chunks:
        by_agency.setdefault(c["agency"], []).append(c)

    # ── ORDER AGENCIES BY AUTHORITY RANK (highest authority first) ────────────
    ordered_agencies = sorted(
        by_agency.keys(),
        key=lambda a: AUTHORITY_RANK.get(a, 99)
    )

    # ── BUILD CONTEXT BLOCK ───────────────────────────────────────────────────
    blocks   = []
    source_n = 1

    for agency in ordered_agencies:
        rank = AUTHORITY_RANK.get(agency, "?")
        blocks.append(f"\n=== AGENCY: {agency} (Authority Level {rank}) ===")

        for c in by_agency[agency]:
            # Best section ID available — first from list, or fallback
            sections = c.get("section_ids", [])
            section  = sections[0] if sections else "§ unknown"

            header = (
                f"[Source {source_n}] {agency} — "
                f"{c.get('source_file', 'unknown')}, {section}, p.{c.get('page', '?')}"
            )
            blocks.append(f"{header}\n{c['text']}")
            source_n += 1

    blocks.append("\n=== END OF RETRIEVED CONTEXT ===")
    return "\n\n".join(blocks)
